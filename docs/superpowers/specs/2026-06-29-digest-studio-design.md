# Subsystem E — Digest Studio

**Date:** 2026-06-29
**Status:** Approved design (pre-implementation)
**Part of:** follow-on to Subsystem D (Admin Panel). Builds on the server-rendered
`/admin` console, the `app_settings`/write-pool plumbing, and the Subsystem C digest.

## Problem

The daily competitor digest email is generated by a **single hardcoded Gemini prompt**
(`geminiNarrate` in `ingestion/src/digest/render.ts`) and sent once at 03:15 by the
ingestion run. The user cannot see what the email will look like before it goes out,
cannot tune the prompt without an SSH/code change, and cannot run more than one digest
(e.g. a different framing, audience, or language) on its own schedule.

Digest Studio adds a self-service admin area to **author digest prompts, preview/test the
email each one produces against real data, and schedule any number of them** — replacing
the single hardcoded send.

## Decisions (locked during brainstorming)

- **Preview uses real data + Gemini.** Preview runs the live `dailyDigest()` and calls
  Gemini with the prompt being edited — a true preview of the next email, not a sample.
- **Send-test button.** Each prompt has a "Send test to me" action that emails the rendered
  preview to the logged-in admin via Resend.
- **The prompt owns the language.** No more EN/MK auto-injection — the prompt text itself
  specifies the language(s) and structure. `geminiNarrate` takes the prompt as an argument.
- **Multiple prompts and multiple schedulers**, created via `＋` buttons. Prompts and
  schedulers are separate, linked entities: a scheduler points at a prompt.
- **In-process scheduler** in the always-on mcp-server (not OS cron / systemd timer).
- **Cutover:** seed the current digest as one prompt + one 07:00 schedule; remove the old
  hardcoded send from the ingestion daily run so nothing double-sends.

## Architecture

```
Browser ──HTTPS──▶ /admin/digests  (server-rendered, admin-gated, CSRF)
   │  Prompts: list / new / edit / preview / send-test / delete
   │  Schedulers: list / new / edit / delete
   ▼
write DB pool (DATABASE_URL) ── digest_prompts, digest_schedules
read  DB pool ──────────────── dailyDigest() data (competitors, sales, ads, social, inv.)

mcp-server process (always on):
   digestScheduler()  every 60s, Europe/Skopje:
       for each enabled schedule due now & not yet run today →
           dailyDigest(db) → renderDigestWithPrompt(digest, prompt.body)
           → sendDigestEmail(to: schedule.recipients ?? digest_recipients)
           → set last_run_on = today   (failure-isolated, logged, recordRun)

@mytime/db  (shared render — moved out of ingestion):
   renderDigestWithPrompt(digest, promptBody) → { subject, html }
   geminiNarrate(digest, promptBody) → string | null   (Gemini, prompt-driven)
   templateDigest(digest) → string                      (deterministic fallback)
```

## 1. Data model (new tables, Drizzle migration `0003`)

```sql
CREATE TABLE digest_prompts (
  id          text PRIMARY KEY,            -- slug, e.g. "daily-default"
  name        text NOT NULL,
  body        text NOT NULL,               -- full Gemini instruction (language included)
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE digest_schedules (
  id          text PRIMARY KEY,            -- slug
  name        text NOT NULL,
  prompt_id   text NOT NULL REFERENCES digest_prompts(id) ON DELETE RESTRICT,
  send_at     text NOT NULL,               -- "HH:MM", interpreted in Europe/Skopje
  recipients  jsonb,                       -- string[] | null → falls back to digest_recipients
  enabled     boolean NOT NULL DEFAULT true,
  last_run_on date,                        -- date the schedule last fired (idempotency)
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now()
);
```

- `ON DELETE RESTRICT`: a prompt referenced by a schedule cannot be deleted until the
  schedule is removed (UI surfaces this as a friendly error).
- Slugs (`id`) are generated from the name (kebab-case, de-duped with a numeric suffix) so
  URLs are stable and readable, consistent with `targets.id`.

### Data-access helpers (`@mytime/db/src/digests-db.ts`)
- `listPrompts(db) / getPrompt(db,id) / upsertPrompt(db,{id?,name,body}) / deletePrompt(db,id)`
- `listSchedules(db) / getSchedule(db,id) / upsertSchedule(db,{...}) / deleteSchedule(db,id)`
- `dueSchedules(db, todayLocal, hhmm)` → enabled schedules with `send_at === hhmm` and
  `last_run_on` IS NULL or `< todayLocal`, joined to their prompt.
- `markScheduleRan(db, id, todayLocal)` → sets `last_run_on`.

## 2. Shared render + send (move into `@mytime/db`)

Because the scheduler runs **inside mcp-server** (a sibling package that must not import from
`ingestion`), both rendering and sending move into `@mytime/db`, which mcp-server already
depends on for `dailyDigest`:

- `@mytime/db/src/digest-render.ts` — `geminiNarrate`, `templateDigest`,
  `renderDigestWithPrompt` (moved from `ingestion/src/digest/render.ts`).
- `@mytime/db/src/digest-send.ts` — `sendDigestEmail` (moved from
  `ingestion/src/digest/send.ts`; the Resend call is unchanged).

Changes:

- `geminiNarrate(digest, promptBody, model?)` — takes the prompt as an argument; the prompt
  body becomes the Gemini `system_instruction`. No language parameter. Returns `null` on any
  failure/missing key (unchanged behaviour).
- `renderDigestWithPrompt(digest, promptBody)` → `{ subject, html, usedFallback }`: calls
  `geminiNarrate`, falls back to `templateDigest(digest)` (one deterministic bilingual block)
  if Gemini returns null, and reports which path was used (so Preview can label a fallback).
  Wraps the body in the existing email HTML shell.
- `sendDigestEmail({subject, html}, to: string[])` — `to` is now required and explicit (the
  caller resolves `schedule.recipients ?? digest_recipients`); no env-list lookup inside.

The MK/EN deterministic `templateDigest` is collapsed to one function that emits both
languages (the prompt now controls AI-language; the fallback just needs to be readable).
`ingestion/src/digest/{render,send}.ts` are deleted after the move.

## 3. Admin page `/admin/digests`

One nav item **Digests**, one list page with two cards:

- **Prompts** — table (Name · Updated · Edit) + `＋ New prompt`.
- **Schedulers** — table (Name · Prompt · Time · Status · Edit) + `＋ New scheduler`.

### Prompt editor `/admin/digests/prompts/:id` (`:id` = `new` for create)
- `name` input, `body` `<textarea>` (monospace, tall), Save / Delete.
- **Preview** (POST `…/preview`): runs `dailyDigest(readDb)` + `renderDigestWithPrompt`,
  returns the email HTML rendered inside a sandboxed `<iframe srcdoc>` below the form, plus
  the computed subject. Surfaces Gemini/data errors inline (no silent fallback in preview —
  if Gemini failed, say so and show the template result labelled as fallback).
- **Send test to me** (POST `…/test`): renders + `sendDigestEmail(to:[admin.email])`;
  flash "Test sent to dragan@mytime.mk".

### Scheduler editor `/admin/digests/schedules/:id`
- `name`, prompt `<select>` (from `digest_prompts`), `send_at` (`<input type=time>`),
  `recipients` (textarea, one email per line; blank = use global Recipients), `enabled`
  checkbox. Save / Delete.

### Routes (added to `mcp-server/src/admin/router.ts`)
```
GET  /digests                          list (Prompts + Schedulers)
GET  /digests/prompts/:id              prompt editor (id="new" → blank)
POST /digests/prompts                  save  (action=save|delete)
POST /digests/prompts/:id/preview      render preview (returns HTML page w/ iframe)
POST /digests/prompts/:id/test         send test email to admin
GET  /digests/schedules/:id            scheduler editor
POST /digests/schedules                save  (action=save|delete)
```
All under `requireAdmin`; every POST carries the per-session CSRF token. Page modules live
in `mcp-server/src/admin/pages/digests.ts` (list + both editors + submit/preview/test
handlers), following the existing `pages/*.ts` `render`/`submit` pattern, extended with the
named handlers the new routes need.

## 4. Scheduler engine (`mcp-server/src/digestScheduler.ts`)

- `startDigestScheduler()` — `setInterval(tick, 60_000)` plus one immediate `tick()` guard.
- `tick()`: compute `nowLocal` in **Europe/Skopje** (`Intl.DateTimeFormat` with
  `timeZone: "Europe/Skopje"`), derive `todayLocal` (YYYY-MM-DD) and `hhmm` (HH:MM). For each
  `dueSchedules(db, todayLocal, hhmm)`: build digest, render with the schedule's prompt, send
  to `schedule.recipients ?? digest_recipients`, `markScheduleRan`, `recordRun(collector:
  "digest:"+id)`. Each schedule is wrapped in try/catch so one failure can't stop the others.
- Started from `server.ts` after `listen(...)`, guarded by `optionalEnv("RESEND_API_KEY")`
  (no key → scheduler logs "disabled" and does not start).
- Idempotency via `last_run_on`: at most one send per schedule per local day, even across
  process restarts. A minute-granularity match means a schedule whose minute is missed
  (process down at exactly HH:MM) is skipped that day — acceptable; documented.

## 5. Cutover

1. **Seed** (migration `0003` data step or `db:seed`): insert prompt `daily-default`
   (name "Daily competitor digest", body = the current EN+MK instruction text) and schedule
   `daily-0700` (prompt_id `daily-default`, send_at "07:00", recipients null, enabled true).
2. **Remove** the digest phase from `ingestion/src/index.ts:258-261` (the
   `runDigestEmail(db)` block + its import) so the ingestion run no longer sends. With
   `render.ts`/`send.ts` moved to `@mytime/db`, delete the remaining
   `ingestion/src/digest/` files (`job.ts`, `cli.ts`) and drop the `digest:email` package
   script. The in-process scheduler is now the sole sender.
3. **Env:** add `GEMINI_API_KEY` and `RESEND_API_KEY` to the mcp-server `.env` on the VPS
   (currently only ingestion has them). Without `GEMINI_API_KEY`, preview/scheduled sends use
   the deterministic template; without `RESEND_API_KEY`, the scheduler stays off and the
   send-test button errors with a clear message.

## 6. Security

- All `/admin/digests*` routes behind the existing Google-domain + admin-role gate; CSRF on
  every POST (reuse `csrfField`/`checkCsrf`).
- Preview renders the email HTML in a **sandboxed iframe** (`sandbox` with no
  `allow-scripts`) so prompt-generated HTML cannot run script in the admin origin.
- `recipients` validated as `@`-bearing addresses server-side; `send_at` validated against
  `^\d{2}:\d{2}$` with 00–23 / 00–59 ranges; prompt `body` length-capped (e.g. 8 KB).
- Test/scheduled sends use the existing Resend send-only key. Secrets remain `.env`-only.

## 7. When changes take effect

- **Prompt edits** — used by the next preview/test immediately, and by any schedule on its
  next fire.
- **Schedule edits** — picked up on the next 60s tick (the scheduler reads the DB each tick;
  no restart needed).
- **Data freshness** — all digests summarize the latest ingested data (refreshed daily at
  03:15); that's why the seeded schedule defaults to 07:00.

## Testing

- **Unit (`@mytime/db`):** `geminiNarrate` builds the request with the given prompt as
  `system_instruction` and returns null on non-OK / missing key (fetch mocked);
  `renderDigestWithPrompt` falls back to `templateDigest` when narrate returns null;
  slug generation + de-dupe; `dueSchedules` matching (enabled, time match, not-run-today,
  across a date boundary); recipients/`send_at` validators.
- **Integration (live DB):** prompt CRUD round-trip; schedule CRUD round-trip; `deletePrompt`
  blocked while referenced; `dueSchedules` returns a seeded due row and `markScheduleRan`
  flips it off for the rest of the day.
- **Scheduler:** a `tick()` with a stubbed clock at the seeded time sends exactly once and
  is idempotent on a second tick the same day (send mocked).
- **Auth/CSRF:** non-admin blocked from `/admin/digests`; POST without CSRF rejected.
- `pnpm -r build` + Biome clean.

## Scope / YAGNI

- No per-recipient unsubscribe, no open/click tracking, no template WYSIWYG — prompt + raw
  HTML preview only.
- Schedules are daily-at-a-time-of-day (HH:MM); no full cron (days-of-week, intervals). If
  needed later, `send_at` can grow into a cron expression without a model change.
- No prompt version history (just `updated_at`); no draft/publish — Save is live.
- Minute-granularity firing; a send missed because the process was down at that exact minute
  is skipped for the day (not retroactively sent).

## Success criteria

1. An admin can create/edit/delete digest prompts at `/admin/digests`, **Preview** renders
   the real next email through Gemini in-page, and **Send test** delivers it to the admin.
2. An admin can create/edit/delete schedulers binding a prompt to a time + recipients; each
   enabled schedule sends once per day at its Europe/Skopje time via the in-process scheduler.
3. The old hardcoded 03:15 send is removed; the seeded `daily-default` prompt + `daily-0700`
   schedule reproduce the existing digest with no double-send.
4. The prompt fully controls language/structure; Gemini failure falls back to the template.
5. Build + Biome clean; auth gate + CSRF + iframe sandboxing verified; secrets never exposed.
